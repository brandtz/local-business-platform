## Integration Flows
### Order Flow
1. Customer checkout → `createOrderPaymentIntent()` creates authorized intent (idempotency key: `order_intent_{orderId}`)
2. Order fulfillment (ready/completed) → `captureOrderPayment()` transitions authorized → captured
3. Order cancellation (before capture) → `voidOrderPayment()` transitions authorized → voided
4. No intent exists → operations return null gracefully

### Booking Flow
1. Booking confirmation with payment → `createBookingPaymentIntent()` creates authorized intent (idempotency key: `booking_intent_{bookingId}`)
2. Booking completed → `captureBookingPayment()` transitions authorized → captured
3. Booking cancellation (before capture) → `voidBookingPayment()` transitions authorized → voided

## Idempotency
- Order intents use `order_intent_{orderId}` as idempotency key
- Booking intents use `booking_intent_{bookingId}` as idempotency key
- Duplicate calls return existing transaction (no double-charge)

## Failure Handling
- Payment failure during intent creation returns transaction with `failed` status
- Capture/void on non-authorized transaction returns existing transaction (no state mutation)
- Operations on non-existent transactions return null (not error)

## E8-S1/S2 Contracts Consumed

- Inherits multi-processor routing and failover from orchestration layer
