## What Changed

- Changed surfaces:
  - `apps/web-customer/src/main.ts` — rewritten to gate Vue app mount on tenant bootstrap resolution. On success: full storefront app with tenant context provided via injection. On failure: minimal error shell with bootstrap failure reason.
  - `apps/web-customer/src/routes.ts` — accepts optional `TenantFrontendContext` parameter. Routes render tenant-aware descriptions when context is provided.
  - New file: `apps/web-customer/src/tenant-context.ts` — bootstrap orchestrator, injection keys, pluggable data source contract, dev/test sessionStorage override, managed preview domain config reader.
  - New file: `apps/web-customer/src/tenant-context.test.ts` — 19 tests covering orchestrator resolution, failure paths, override parsing, and domain config reading.
  - Updated: `apps/web-customer/src/routes.test.ts` — added test for routes with tenant context parameter.

- New or updated contracts:
  - `TENANT_CONTEXT_KEY: InjectionKey<TenantFrontendContext>` — available when bootstrap resolves. Components inject this for tenant-aware rendering.
  - `TENANT_BOOTSTRAP_RESULT_KEY: InjectionKey<BootstrapResult>` — always available. Components inject this to inspect bootstrap phase/failure.
  - `TenantBootstrapDataSource` — `() => Promise<{ tenants, tenantConfig }>` — pluggable data fetcher. E4-S2-T3 will provide the API-backed implementation.
  - `TenantBootstrapOptions` — `{ host, managedPreviewDomains, fetchData }` — bootstrap config.
  - `executeTenantBootstrap(options)` — async orchestrator that fetches data then calls `resolveBootstrap()`.

- New config, env vars, events, or migrations:
  - `VITE_MANAGED_PREVIEW_DOMAINS` — comma-separated list of managed preview domain suffixes (default: `preview.localhost`).
  - `__platform_test_web_customer_tenant_bootstrap__` — sessionStorage key for dev/test bootstrap data override (follows same pattern as auth-state override).

## Assumptions and Risks

- Assumptions made:
  - The `TenantBootstrapDataSource` will be replaced by a real API call in E4-S2-T3. Currently uses dev fallback (sessionStorage override or empty tenant list).
  - Dev mode will fail with `tenant-not-found` unless a sessionStorage override is set — this is intentional to surface the bootstrap flow clearly.
  - `resolveBootstrap()` from tenant-bootstrap.ts is the canonical resolution function — this task wraps it with async data fetching, not replaces it.

- Risks introduced or still open:
  - In dev without sessionStorage override, the app always mounts the failure shell. Developers must set the override or E4-S2-T3 must provide the API data source.
  - Injection keys are Symbols — if tree-shaking splits the module, keys won't match across bundles (mitigated by same-package import).

- Invariants changed: no
