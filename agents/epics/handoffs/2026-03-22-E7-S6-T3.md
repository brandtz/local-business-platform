## Gating Locations in Frontend

### Web Customer (Storefront)

| Route Pattern | Blocked When | Behavior |
|---|---|---|
| /book, /account/bookings, /account/bookings/:id | ordering-only mode | `isRouteBlockedByOperatingMode()` returns true |
| /order, /account/orders, /account/orders/:id | booking-only mode | `isRouteBlockedByOperatingMode()` returns true |
| /cart, /checkout | booking-only mode | `isRouteBlockedByOperatingMode()` returns true |

### Web Admin

| Route | Blocked When | Behavior |
|---|---|---|
| /ordering | booking-only mode | `evaluateAdminOperatingModeRouteGuard()` returns `{ allowed: false, reason: "module-disabled" }` |
| /bookings | ordering-only mode | `evaluateAdminOperatingModeRouteGuard()` returns `{ allowed: false, reason: "module-disabled" }` |

## Shared vs Mode-Specific Admin Behavior

### Shared Admin Sections (available in all modes)
- Dashboard, Catalog, Content, Operations, Users, Settings, Activity Log

### Mode-Specific Admin Sections
- **Ordering mode**: Ordering section visible
- **Booking mode**: Bookings section visible
- **Hybrid mode**: Both Ordering and Bookings sections visible

## Integration with Existing Module Gating

The operating mode gating builds on top of the existing module-based gating from E3/E4/E5:
- `filterNavigationByModules()` (web-customer) already hides disabled module routes
- `filterNavigationForContext()` (web-admin) already filters by role + module
- The new operating mode helpers provide higher-level semantics (mode → flow → route mapping)
