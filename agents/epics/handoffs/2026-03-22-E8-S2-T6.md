## Failover Behavior
1. **Trigger**: When primary processor's `createIntent()` fails
2. **Condition**: `ProcessorRoutingConfig.failoverEnabled` must be true AND a secondary active connection must exist
3. **Process**:
   a. Log `FailoverEvent` with primary provider, failover provider, error message, reference info
   b. Attempt `createIntent()` on secondary processor
   c. If secondary succeeds: emit `failover_triggered` + `intent_created` audit events, return authorized transaction
   d. If secondary also fails: create `failed` transaction record, emit `payment_failed` audit event
4. **No failover**: If failover disabled or no secondary connection, primary failure creates `failed` transaction directly

## E8-S6 Alert Pipeline Integration
- `getFailoverEventLog()` returns readonly array of all failover events
- `clearFailoverEventLog()` resets the log (for testing/processing)
- E8-S6 should poll or subscribe to this log for operational alerting
- Each event includes enough context to identify the affected tenant, processors, and transaction
