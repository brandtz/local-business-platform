## Provider-Neutral Connection Status Model
- Status transitions: `inactive → verifying → active`, `active → suspended`, `active → verifying` (re-verify), `suspended → verifying` (recovery), `suspended → inactive` (deactivate)
- Verification: credentials decrypted, passed to provider adapter, status updated based on result
- Health check: active connections pinged via adapter, suspended on failure
- Deactivation: admin can deactivate from active/suspended

## Secret-Handling Rules
- All service methods return `AdminPaymentConnectionSummary` (sanitized, no credentials)
- Credentials decrypted only for verification/health check, never returned
- Adapter pattern ensures no provider SDK calls leak into domain logic
