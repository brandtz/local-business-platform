## Dependency Outputs
- **Service**: `PaymentTransactionService` — main entry point for all payment transaction operations
- **Errors**: `PaymentTransactionNotFoundError`, `PaymentTransactionTransitionError`, `PaymentTransactionValidationError`, `PaymentProcessorError`
- **Audit**: Every state change emits a `PaymentAuditEvent` with action, actor, reason, amounts
- **Idempotency**: Duplicate idempotency keys return existing transaction (no re-processing)

## Transaction State Mapping
- `createPaymentIntent()` → `authorized` (on success) or `failed` (on provider failure)
- `capturePayment()` → `captured` (authorized → captured)
- `voidPayment()` → `voided` (authorized → voided)
- `processRefund()` → `partially_refunded` or `refunded` (captured/partially_refunded → refunded)

## Refund Audit Contract
- Refund requires `actorId` (non-empty) and `reason` (non-empty) — enforced with validation errors
- Emits `refund_initiated` audit event BEFORE provider call (with actor, reason, amount)
- Emits `refund_completed` audit event AFTER successful provider refund
- All audit events include `tenantId`, `transactionId`, `provider`, `newStatus`

## Multi-Processor Routing Rules (E8-S2-T5)
- `resolveConnection()`: checks `ProcessorRoutingConfig` for preferred primary provider, falls back to first active connection
- `resolveFailoverConnection()`: returns alternative active connection if failover enabled
- Refunds always route to the original capture processor (stored in `connectionId`)

## Failover Behavior (E8-S2-T6)
- When primary `createIntent()` fails and failover is enabled, attempts secondary processor
- `FailoverEvent` logged with primary/failover providers, error message, reference info
- `getFailoverEventLog()` exposes events for E8-S6 alert pipeline consumption
- If both processors fail, transaction is recorded as `failed`
