## Changed Surfaces

### Types Package (in booking.ts)
- `BookingSlotQuery` — Query parameters: tenantId, locationId, serviceId, staffId (optional), startDate, endDate
- `StorefrontBookingSlot` — Public slot (startTime, endTime only — no staff identity)
- `StorefrontBookingSlotsResponse` — Storefront response (serviceId, date, slots[])
- `AdminBookingSlot` — Admin slot (startTime, endTime, staffId, staffName)
- `AdminBookingSlotsResponse` — Admin response (serviceId, date, slots[])
- `BookingSlotValidationError` — Typed validation error union
- `BookingSlotValidationResult` — Validation result type
- `validateBookingSlotQuery()` — Validates query parameters (required fields, date formats, range limits)

### API Contracts
- **`platform/apps/api/src/booking/booking-slot-api-contracts.ts`** — New contract module:
  - `assertValidBookingSlotQuery()` — Runtime assertion for request validation
  - `toStorefrontBookingSlotsResponse()` — Maps `SlotComputationResult` to public format (deduplicates by time, strips staff identity)
  - `toAdminBookingSlotsResponse()` — Maps `SlotComputationResult` to admin format (preserves all slots with staff details)
  - `assertValidStorefrontResponse()` — Validates storefront response shape
  - `assertValidAdminResponse()` — Validates admin response shape (requires staffId and staffName on each slot)

## Query Parameter Validation Rules
- `serviceId`: Required, non-empty string
- `locationId`: Required, non-empty string
- `startDate`: Required, YYYY-MM-DD format
- `endDate`: Required, YYYY-MM-DD format
- Date range: endDate must be ≥ startDate, max 31 days
- `staffId`: Optional, but if provided must be non-empty

## Storefront vs Admin Differences
| Aspect | Storefront | Admin |
|--------|-----------|-------|
| Staff identity | Stripped | Included (staffId, staffName) |
| Slot deduplication | Yes (same time = 1 slot) | No (shows all staff options) |
| Use case | Customer booking flow | Scheduling/management |
