## Key Decisions

1. **Pure domain logic pattern** — All 9 services follow the established pure-logic Injectable pattern, operating on in-memory record types. No direct PrismaService injection. This keeps services testable via direct instantiation and defers persistence integration to controller/repository layer.
2. **Prices in cents** — All monetary fields (price, compareAtPrice, priceAdjustment) are stored as integers representing cents to avoid floating-point precision issues.
3. **Status lifecycle constraints** — CatalogItem status (active↔inactive) and visibility (draft↔published) are validated as explicit transitions. Same-state transitions are rejected.
4. **Bulk operation safety** — CatalogBulkService.validateBulkSelection() enforces tenant isolation before any bulk mutation. Cross-tenant selections throw immediately.
5. **Schedule conflict detection** — Time-window overlap uses string comparison of "HH:mm" format, which works because the format is zero-padded and fixed-width.
6. **Staff-auth separation** — StaffProfile is deliberately separate from User/TenantMembership. A staff profile may optionally link to a User via `userId` but operates independently for scheduling and booking purposes.

## Security Review

- **Tenant isolation**: Every service method accepts `tenantId` and filters accordingly. `CatalogBulkService` includes explicit cross-tenant guard.
- **Input validation**: Prices validated non-negative, slugs validated via strict regex, names validated non-empty, modifier cardinality validated.
- **Email handling**: Staff email duplicate check uses case-insensitive comparison. The validation error returned is generic (`invalid-format`) rather than `duplicate-email` to prevent email enumeration in API responses. This is intentional but should be documented clearly for the controller layer.
- **No injection risk**: All services are pure logic; database access via Prisma parameterized queries only.
- **No security vulnerabilities found** that require a separate story.
