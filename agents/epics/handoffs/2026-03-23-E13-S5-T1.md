# Handoff: E13-S5-T1 — Admin Shell and Navigation

## Task

Build the AdminLayout component with sidebar, header, and impersonation banner. Define admin routes with auth guards and wire up module-gated navigation.

## Admin Route Tree with Auth Guards and Module Gating

Routes are defined in `routes.ts`. Every admin route carries `meta.requiresAuth: true`, which is checked by the global navigation guard before allowing access. Child routes are nested under the `/admin` path prefix and rendered inside the AdminLayout shell.

```
/admin
  ├── /admin/dashboard        (meta.requiresAuth: true)
  ├── /admin/profile           (meta.requiresAuth: true)
  ├── /admin/payments          (meta.requiresAuth: true)
  ├── /admin/users             (meta.requiresAuth: true)
  ├── /admin/activity          (meta.requiresAuth: true)
/admin/login                    (meta.requiresAuth: false)
```

The navigation guard reads `to.meta.requiresAuth` and redirects unauthenticated users to `/admin/login`.

## Sidebar Navigation Data Structure with Gating Keys

The sidebar builder in `main.ts` maps route entries to `AdminSidebarItem[]`. Each item includes a `gatingKey` that corresponds to a module identifier returned by the SDK. Items whose gating key is not present in the active module set are filtered out before rendering.

```ts
interface AdminSidebarItem {
  label: string;
  icon: string;
  route: string;
  gatingKey: string; // e.g. "analytics", "orders", "payments", "users", "audit"
}
```

Module gating is performed by `filterNavigationForContext()` from `module-navigation.ts`, which accepts the full sidebar item list and the tenant's enabled modules, returning only the items the current tenant may access.
