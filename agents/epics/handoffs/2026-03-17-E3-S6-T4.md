## Assumptions and Risks

- Assumptions made:
  - filter criteria field names matching operational summary field names is the correct design choice for consistency
  - text search using `includes()` on lowercased strings is sufficient for initial search semantics
  - the filter function is a pure function on the shared types layer so both backend and frontend can reuse it
- Risks introduced or still open:
  - text search is linear scan; will need optimization (indexed search or database query) when tenant count grows
  - no pagination support yet; large result sets will need cursor-based or offset pagination in a follow-on task
