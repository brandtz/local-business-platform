## Cache Strategy

### Cache Key Structure
```
slots:{tenantId}:{locationId}:{serviceId}:{date}[:{staffId}]
```
- Tenant-scoped: prevents cross-tenant cache pollution
- Granular: per-date, per-service for targeted invalidation
- Redis-compatible: ready for production upgrade

### TTL Policy
- Default: 5 minutes (300 seconds)
- Configurable via constructor for testing
- Balances freshness with query performance

### Invalidation Triggers
| Trigger | Scope | Description |
|---------|-------|-------------|
| `booking-created` | tenant + location + date | New booking blocks a slot; invalidate that date |
| `booking-cancelled` | tenant + location + date | Cancelled booking frees a slot; invalidate that date |
| `schedule-changed` | tenant + location (all dates) | Staff schedule change affects all future slots |
| `blackout-updated` | tenant + location (all dates) | Blackout change affects multiple dates |
| `hours-changed` | tenant + location (all dates) | Location hours change affects all dates |

### Memory Management
- Max 1000 entries per tenant
- When limit reached, oldest 10% of entries evicted (LRU-like)
- `evictExpired()` available for periodic cleanup

### Production Upgrade Path
The in-memory implementation is suitable for single-instance deployments. For multi-instance production:
1. Replace `Map` with Redis using same key structure (`buildSlotCacheKey()`)
2. Use Redis `SET ... EX` for TTL
3. Use Redis key patterns (`slots:tenant-1:loc-1:*`) for invalidation
4. The `SlotCacheInvalidationTrigger` events can be published via domain events (from `domain-events.ts`)

## Caching Assumptions
1. Slot computation is deterministic for given inputs — caching is safe
2. 5-minute TTL is acceptable for booking use cases (slight staleness tolerated)
3. Immediate invalidation on booking mutations ensures no double-booking via stale cache
4. Single-instance in-memory cache is acceptable for initial deployment
5. Cache keys are tenant-scoped — no cross-tenant leakage possible
