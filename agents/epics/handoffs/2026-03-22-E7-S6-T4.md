## Operating Mode Overview

Tenants can operate in one of three modes, determined by their enabled module set:

| Mode | Condition | Description |
|---|---|---|
| **Ordering** | `ordering` module enabled, `bookings` module disabled | Cart, checkout, and order management are active. Booking flows are disabled. |
| **Booking** | `bookings` module enabled, `ordering` module disabled | Appointment booking and scheduling are active. Cart, checkout, and order flows are disabled. |
| **Hybrid** | Both `ordering` and `bookings` modules enabled | All transaction flows are active: cart, checkout, orders, and bookings. |

### Booking Mode Only

### Hybrid Mode
- **Both Ordering and Bookings Sections**: Full access to all order and booking management capabilities

## How Mode Changes Work

- Operating mode is **derived** from the tenant's enabled module set — it is not a separate configuration toggle.
- To switch modes, an admin changes the enabled modules for their tenant.
- Mode changes take **immediate effect** across both API and frontend layers.
- No restart or re-deployment is required.

### Mode Transition Examples

| From | Action | To |
|---|---|---|
| Ordering | Enable `bookings` module | Hybrid |
| Booking | Enable `ordering` module | Hybrid |
| Hybrid | Disable `ordering` module | Booking |
| Hybrid | Disable `bookings` module | Ordering |

## Implications of Mode Changes for Existing Data

- **Existing orders** remain accessible even if the ordering module is later disabled. Historical data is preserved.
- **Existing bookings** remain accessible even if the bookings module is later disabled. Historical data is preserved.
- Disabling a module prevents **new** transactions of that type but does not delete or hide existing records.
- Customer order/booking history in the account area follows the same gating rules — history tabs for disabled modules are hidden.

## Enforcement Layers

### API Layer (Backend)
- `OperatingModeEnforcementService.requireFlow(context, flow)` throws `OperatingModeError` for blocked flows
- Error messages include the flow name and operating mode: e.g., "Cart operations are not available in booking operating mode."
- Enforcement is per-tenant, per-request (stateless)

### Frontend Layer (Customer Storefront)
- Navigation items for disabled flows are hidden
- Route guards redirect to home for disabled flow routes
- `isRouteBlockedByOperatingMode()` checks route paths against blocked flows
- Direct URL access to disabled routes is handled gracefully (redirect)

### Frontend Layer (Admin Portal)
- Navigation sections for disabled flows are hidden from sidebar
- Route guards return `{ allowed: false, reason: "module-disabled" }` for disabled flow routes
- `evaluateAdminOperatingModeRouteGuard()` provides mode-aware access decisions
- Admin shared sections (dashboard, catalog, content, operations, users, settings, audit) are always visible
