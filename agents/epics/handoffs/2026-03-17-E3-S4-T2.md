- New or updated contracts:
  - `DomainVerificationCheckAdapter` — port interface: `check(hostname, tenantId) → { evidence, passed }`. Downstream DNS implementations satisfy this without coupling workflow to a provider.
  - `DomainVerificationWorkflowService` — orchestrates verification state transitions:
    - `verify()`: runs adapter check, transitions `pending → verified|failed`, `failed → verified`, `verified → failed`, handles same-state no-ops
    - `deny()`: transitions any non-denied state to `denied`
  - `DomainVerificationWorkflowResult` — discriminated union: `verified`, `failed`, `denied`, or `no-transition`
  - `DomainVerificationCheckResult` — adapter output with `CustomDomainVerificationEvidence` and boolean `passed`
  - Same-state re-checks (e.g. `failed → failed`) return `no-transition` instead of throwing

## Assumptions and Risks

- Assumptions made:
  - the port/adapter pattern is the correct abstraction for DNS verification without coupling to a provider
  - same-state no-op (failed → failed) is better handled as `no-transition` than as an error
  - evidence recording is the adapter's responsibility; the workflow stores whatever the adapter returns
- Risks introduced or still open:
  - no persistence layer yet; the workflow returns decisions but does not persist state changes
  - a concrete DNS adapter must be injected before the workflow can be used in production
