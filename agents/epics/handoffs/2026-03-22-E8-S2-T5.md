## Multi-Processor Routing Rules
1. **Primary preference**: `ProcessorRoutingConfig.primaryProvider` determines which active connection is preferred
2. **Resolution**: `resolveConnection(tenantId)` checks routing config first, falls back to first active connection
3. **Failover resolution**: `resolveFailoverConnection(tenantId, excludeProvider)` returns alternate active connection if failover enabled
4. **Refund routing**: Refunds always target the original capture processor (stored in `PaymentTransactionRecord.connectionId`) — not the current routing config
5. **No active connection**: Throws `PaymentProcessorError` if no active connections exist

## Routing Algorithm
```
1. Get all active connections for tenant
2. If routing config exists:
   a. Find connection matching primaryProvider
   b. If found, use it
3. If no config or primary not active, use first active connection
4. For failover: find active connection with different provider (if failoverEnabled)
```

## Security Constraints
- Routing config is tenant-scoped
- Refund never crosses processor boundaries (always original capture connection)
- Connection credentials accessed only through `decryptCredentials()` at point of use
