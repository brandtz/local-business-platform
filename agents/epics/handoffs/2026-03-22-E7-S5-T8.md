## Cross-Tenant Identity Model

### Design Principles

1. **Shared Identity, Separate Profiles**: One person (same email) gets a single `userId` across all tenants, but a separate `CustomerProfile` record per tenant.

2. **Tenant Isolation Is Hard**: All queries — order history, booking history, addresses, payment methods, loyalty, notification preferences — are scoped to `(tenantId, customerId)`. A customer's data in Tenant A is NEVER returned when querying Tenant B.

3. **Registration Per Tenant**: Same email can register separately in multiple tenants. Each registration creates a new CustomerProfile with a unique `id` but the same `userId`.

### Identity Mapping

```
User (shared identity)
├── userId: "user-alice@example.com"
├── email: "alice@example.com"
├── CustomerProfile (Tenant A)
│   ├── id: "cust-1"
│   ├── tenantId: "tenant-a"
│   ├── orders, bookings, addresses, etc.
│   └── loyalty (separate points/tier)
└── CustomerProfile (Tenant B)
    ├── id: "cust-2"
    ├── tenantId: "tenant-b"
    ├── orders, bookings, addresses, etc.
    └── loyalty (separate points/tier)
```

### Helper Functions

- `isCustomerInTenant(profile, tenantId)` — Primary tenant isolation check for customer operations
- `getProfileForTenant(mapping, tenantId)` — Extracts the profile for a specific tenant from the cross-tenant mapping
- `getCrossTenantIdentity(userId, tenantNameResolver)` — Returns the full mapping of tenant profiles for a user (used for admin/platform views, NOT for customer data access)

### Security Invariants

- Customer-facing APIs always receive `tenantId` from the resolved request context (host-based), never from the client
- History query methods enforce `tenantId` + `customerId` in every filter
- Cross-tenant identity mapping is NOT exposed to the customer — it's an internal platform model
- No API endpoint returns data across tenants to a customer
