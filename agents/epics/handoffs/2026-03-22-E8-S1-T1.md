## Encrypted Config Contract
- Credentials stored as three fields: `encryptedCredentials` (AES-256-GCM ciphertext hex), `credentialsIv` (IV hex), `credentialsTag` (auth tag hex)
- Encryption key: `PAYMENT_ENCRYPTION_KEY` env var, 32-byte hex string
- Credentials are JSON-serialized before encryption
- Provider-specific credential shapes: `StripeCredentials` (publishableKey, secretKey), `SquareCredentials` (applicationId, accessToken, locationId)

## Secret-Handling Rules
- `paymentConnectionSecretFields` lists all fields that must be stripped from API responses
- `sanitizePaymentConnection()` is the canonical function for stripping secrets
- Tests verify encrypted credentials never appear in any sanitized output

## Security Test Evidence
- Encryption: credentials are not stored in plaintext (ciphertext test), roundtrip decrypt verified
- No-leak: sanitized views verified to exclude all `paymentConnectionSecretFields`
- Uniqueness: duplicate provider per tenant throws `PaymentConnectionDuplicateError`
