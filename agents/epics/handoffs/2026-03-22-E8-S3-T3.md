## Idempotency Rules
1. **Already at target state**: Event marked as `processed` with no state change, no audit event
2. **Terminal state**: Event marked as `skipped` — terminal states (voided, refunded, failed) never regress
3. **Out-of-order**: If transaction is "ahead" of the target (e.g., already captured when authorized event arrives), event is `skipped`
4. **Duplicate processing**: Only one audit event per successful reconciliation

## Replay Procedure
1. Call `replayEvent(eventId)` with the stored event ID
2. Service retrieves the stored event and re-parses the raw payload
3. `processEvent()` is called with the re-parsed payload
4. All idempotency checks apply — replay of already-processed events is safe
5. Returns `{ success: true }` on successful replay, `{ success: false, error }` on failure

## Audit Events
- Reconciliation emits a `PaymentAuditEvent` with:
  - `action`: mapped from target status (`captured`, `voided`, `refund_completed`, `payment_failed`)
  - `reason`: `"Webhook reconciliation: <eventType>"`
  - `previousStatus`: transaction status before update
  - `newStatus`: target status after update
  - `metadata`: `{ webhookEventId, providerEventId }`
